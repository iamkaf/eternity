# Architecture Decision Records

> Each ADR captures a significant technical decision, the options considered, and why we chose what we did. These are extracted from the PRDs for quick reference.

## ADR-001: Entity Data Format — JSON over TOML

**Date**: 2026-04-16
**Status**: Accepted
**Context**: Entity data (actors, items, maps) needs a serialization format. Candidates: JSON, TOML, YAML.

| Option | Pros | Cons |
|---|---|---|
| **JSON** | Native JS parsing, programmatic generation, clean diffs with consistent indent | No comments, verbose for hand-editing |
| TOML | Human-readable, comments, multi-line strings | No native JS parser (needs library), harder to generate programmatically |
| YAML | Human-readable, comments | Whitespace-sensitive, parsing ambiguities, security concerns |

**Decision**: JSON for all entity data. TOML for config files (`project.toml`, `config.toml`) that humans edit directly.

**Rationale**: Entity data is generated by the editor — users rarely hand-edit it. JSON's native parsing, `JSON.stringify` round-tripping, and Git diff friendliness outweigh TOML's readability advantages for this use case.

## ADR-002: ECS Library — Miniplex over bitECS

**Date**: 2026-04-16
**Status**: Accepted
**Context**: The engine needs an ECS. Candidates: Miniplex, bitECS, custom implementation.

| Factor | Miniplex | bitECS |
|---|---|---|
| Data model | Object-based (plain JS objects) | Typed arrays (numeric IDs) |
| Serialization | `JSON.stringify()` works | Manual typed array serialization |
| React bindings | First-class (`createECS`, `useArchetype`) | None |
| Performance ceiling | Hundreds-to-thousands of entities | Tens of thousands |

**Decision**: Miniplex.

**Rationale**: RPG games rarely exceed a few hundred active entities. Miniplex's object-based model aligns with the "single serializable state" architecture, and its React bindings power the editor's entity inspector without adapter code.

## ADR-003: Scripting Sandbox — Web Workers + Comlink

**Date**: 2026-04-16
**Status**: Accepted
**Context**: User-authored scripts must be isolated from Node.js/Electron APIs. Candidates: Web Workers, QuickJS (WASM), V8 Isolates.

| Factor | Web Workers + Comlink | QuickJS (WASM) | V8 Isolates |
|---|---|---|---|
| Cross-platform | ✅ Electron + Web | ✅ Everywhere | ❌ Electron only |
| TypeScript support | ✅ Native runtime | ⚠️ Pre-compilation required | ✅ Native |
| Debugging | ✅ DevTools worker debugging | ❌ Limited | ✅ DevTools |
| API bridging | ✅ Comlink proxies (~2KB) | ⚠️ Manual | ⚠️ Manual |

**Decision**: Web Workers with Google's Comlink library.

**Rationale**: Works identically in Electron and web exports. Comlink makes cross-worker calls look like local async functions. Combined with CSP headers (`connect-src 'none'`), it provides sufficient isolation for game scripting. V8 Isolates were rejected because they don't work in web export. QuickJS was rejected due to TypeScript compilation complexity and lack of DevTools debugging.

## ADR-004: Visual Scripting UI — React Flow

**Date**: 2026-04-16
**Status**: Accepted
**Context**: The visual node graph editor needs a React-compatible library. Candidates: React Flow, Rete.js, LiteGraph.js.

| Factor | React Flow | Rete.js | LiteGraph.js |
|---|---|---|---|
| Framework | React-native | Framework-agnostic | Canvas-based |
| TypeScript | First-class | First-class | Manual type defs |
| Custom nodes | React components | Plugin system | Canvas drawing |
| Ecosystem | Industry standard, large community | Smaller community | Legacy, stable |

**Decision**: React Flow.

**Rationale**: Eternity's editor is React-based. React Flow provides first-class React integration, TypeScript types, and custom node rendering — letting us embed RPG-specific UI (dialogue previews, item pickers) inside graph nodes. Rete.js was a close second but its framework-agnostic approach adds unnecessary abstraction for a React-only editor.

## ADR-005: Monorepo Tooling — pnpm + Turborepo

**Date**: 2026-04-16
**Status**: Accepted
**Context**: The project is a monorepo with ~10 packages. Needs dependency management and task orchestration. Candidates: pnpm + Turborepo, pnpm + Nx, yarn + Lerna.

| Factor | pnpm + Turborepo | pnpm + Nx |
|---|---|---|
| Complexity | Low | High (steeper learning curve) |
| Setup time | Minutes | Hours |
| Features | Task orchestration, caching | Dependency graph, code gen, CI agents |
| Best for | Small-to-medium teams | Large teams, complex monorepos |

**Decision**: pnpm workspaces + Turborepo.

**Rationale**: Eternity is a single-developer or small-team project. Turborepo's simplicity and fast caching are sufficient. Nx's enterprise features (dependency graph enforcement, distributed CI agents) would be overkill and add maintenance overhead.

## ADR-006: Editor UI Contract — json-render Catalog

**Date**: 2026-04-16
**Status**: Accepted
**Context**: Editor panels need a consistent pattern that plugins can extend. Options: raw React component sharing, json-render catalog, custom JSON schema renderer.

**Decision**: Vercel's json-render with `defineCatalog` + `defineRegistry`.

**Rationale**: Plugins must extend the editor UI without shipping raw React components (which would couple them to the editor's React version and internal APIs). json-render's catalog pattern constrains all panels to a Zod-schema-validated JSON tree. Plugins register catalog entries (abstract definitions) and registry entries (React implementations) — the editor shell renders them without knowing about plugin components at compile time.

## ADR-007: WebGPU — Prefer, Don't Require

**Date**: 2026-04-16
**Status**: Accepted
**Context**: PixiJS v8 supports both WebGPU and WebGL. Should we require WebGPU?

**Decision**: WebGPU preferred, WebGL fallback supported.

**Rationale**: PixiJS v8 handles this automatically — it detects WebGPU availability and falls back to WebGL. Requiring WebGPU would gate users on GPU generation, which is unacceptable for an RPG creation tool targeting hobbyist developers. The performance delta between WebGPU and WebGL is negligible for 2D tile-based games.

## ADR-008: Plugin Distribution — Manual (No Registry)

**Date**: 2026-04-16
**Status**: Accepted
**Context**: How do users install plugins? Options: centralized registry (npm-style), manual download, Git submodules.

**Decision**: Manual download. Plugins are zip/tar archives placed in a `plugins/` directory.

**Rationale**: A plugin registry requires infrastructure (hosting, moderation, search, versioning, security scanning) that is premature for v1. Manual distribution is simple, transparent, and unblocks the plugin ecosystem without infrastructure investment. Can be upgraded to a registry later without changing the plugin format.

## ADR-009: Multiplayer Architecture — Relay Server

**Date**: 2026-04-16
**Status**: Accepted
**Context**: Multiplayer needs a networking model. Options: peer-to-peer, relay (authoritative server), hybrid.

**Decision**: Relay-based with authoritative server.

**Rationale**: Authoritative server prevents cheating — all state mutations are validated server-side. Simplifies NAT traversal (clients connect to one known server, not to each other). The server runs the same game engine headlessly via `HeadlessPlatform`, so there's no separate server codebase to maintain.

## ADR-010: Server Export — Node.js SEA

**Date**: 2026-04-16
**Status**: Accepted
**Context**: Multiplayer servers need to be distributed. Options: Docker container, Node.js SEA binary, traditional `node server.js` with instructions.

**Decision**: Node.js Single Executable Application (SEA) via `--build-sea`.

**Rationale**: Target audience is hobbyist game developers, not DevOps engineers. A single binary is dramatically simpler than a Docker container. No runtime dependencies — download, run, done. Project data is embedded in the binary via `sea.getAsset()`.

## ADR-011: Audio Model — 4-Channel (BGM/BGS/ME/SE)

**Date**: 2026-04-16
**Status**: Accepted
**Context**: How to organize audio playback. Options: flat channel system, RPG Maker 4-channel model, general audio bus system.

**Decision**: RPG Maker-compatible 4-channel model (BGM, BGS, ME, SE).

**Rationale**: Users migrating from RPG Maker understand these categories immediately. The model is simple, covers all RPG audio use cases, and maps cleanly to the Web Audio API. A general bus system would be more flexible but harder to use for the target audience.

## ADR-012: Export Targets — Plugin Architecture

**Date**: 2026-04-16
**Status**: Accepted
**Context**: How to support multiple export platforms. Options: hard-coded per-platform logic, plugin-based `ExportTarget` interface.

**Decision**: Pluggable `ExportTarget` interface. Each platform is a plugin.

**Rationale**: New platforms (Tauri, mobile, console) can be added without modifying the core export pipeline. Third-party export targets are first-class. The export dialog auto-generates its config form from the target's Zod `configSchema`, maintaining consistency with the Database Editor's schema-driven approach.

## ADR-013: Desktop Export — Electron First, Tauri Future

**Date**: 2026-04-16
**Status**: Accepted
**Context**: Which desktop shell for exported games? Options: Electron (same as editor), Tauri (lighter), NW.js (RPG Maker's choice).

**Decision**: Electron first. Tauri as a future export target (not editor replacement).

**Rationale**: Electron export dog-foods the same shell as the editor — maximum confidence in compatibility. Tauri v2 offers massive advantages (3-10MB vs 100-200MB bundles, native mobile) but requires a Rust backend, which is a different paradigm. The PAL architecture makes adding `TauriPlatform` straightforward when the time comes. NW.js was rejected due to smaller community and fewer modern tooling updates.

## ADR-014: Asset Optimization — WebP + Opus

**Date**: 2026-04-16
**Status**: Accepted
**Context**: Exported games need optimized assets. What formats to target?

**Decision**: PNG → WebP for textures (25-35% reduction), audio → Opus (best quality-per-byte).

**Rationale**: WebP is supported by all modern browsers and PixiJS v8 loads it natively. Lossless mode preserves pixel-perfect art. Opus has the best quality-per-byte ratio of any web-compatible audio codec. Both are optional optimizations — users can skip them for faster builds.
